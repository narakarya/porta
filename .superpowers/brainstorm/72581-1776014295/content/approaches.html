Tunneling Architecture — 3 Approaches

Pilih struktur yang paling masuk akal untuk bagaimana config dibagi antara Global Settings dan App Settings

A

Global-light, App decides mode

Global hanya simpan cloudflared status + Cloudflare token (opsional). Di level app, user pilih mode (quick vs named), masukkan subdomains. Paling fleksibel per-app.

Pros

  • App A bisa quick tunnel, App B bisa named tunnel
  • Sedikit global config

Cons

  • User harus set mode di tiap app
  • Token tampil di tiap app yang pakai named tunnel
⚙️ Global Settings → Tunneling
cloudflared installed
Cloudflare API Token (optional)
📱 App Settings → Tunneling
Quick Named
Subdomains: api, admin
Connect
B

Global defines provider + default domain, App hanya routing

Global set cloudflared, mode (quick/named), token, dan default domain. App tinggal set subdomain list dan toggle on/off. Mode diwarisi dari global.

Pros

  • App settings sangat bersih — hanya subdomain + toggle
  • Token & mode diatur sekali di global
  • Mudah dipahami user baru

Cons

  • Semua app pakai mode yang sama
  • Kurang fleksibel (tapi bisa di-override nanti)
⚙️ Global Settings → Tunneling
cloudflared installed
Quick Tunnel Named Tunnel
Default domain: example.com
API Token
📱 App Settings → Tunneling
Enable Tunnel ●
Subdomains: api, admin + Add
→ api.example.com, admin.example.com
C

Layered — Global default, App bisa override

Global set default mode + domain. App bisa ikut global atau override mode/domain sendiri. Paling fleksibel, tapi UI paling kompleks.

Pros

  • Mendukung per-workspace domain secara natural
  • Bisa mix quick dan named tunnel per-app

Cons

  • UI lebih complex (inherit vs override)
  • Over-engineered untuk kebutuhan sekarang
⚙️ Global Settings → Tunneling
Default mode: Quick / Named
Default domain: example.com
API Token
📱 App Settings → Tunneling
Inherit global Override
Subdomains
Override: mode, domain

💡 Rekomendasi saya: B — bersih, konsisten, dan model data-nya tetap bisa diperluas ke per-workspace nanti tanpa UI jadi rumit sekarang.
